# 安全与威胁模型

简体中文 · [English](./SECURITY.en.md)

claude-self-goal 会往一个正在运行的 Claude Code 会话里注入输入。用它之前，先弄清楚这意味着什么。

## 它是什么

一个本地自动化 wrapper。它把 `/goal <条件>` 这行字送进你已经在跑的 Claude Code 会话，按会话类型走三条通道之一：tmux 会话用 `tmux send-keys`，普通交互式会话用 `TIOCSTI`，daemon 后台会话用 `claude attach`。用途是让一个自主 agent 在你控制的机器上，给它自己的会话（或你自己的另一个后台会话）设目标。

## 它不是什么

它不是在利用 Claude Code 的 bug，也不是在钻内核漏洞——用的都是有文档、有意提供的接口。但它确实是往会话注入输入，这种能力天生就是双刃的，两条通道各有各的分量：

`TIOCSTI` 会把字节塞进某个终端的输入队列，效果等同于有人在那儿打字。有 root（`CAP_SYS_ADMIN`）、只要内核和容器策略不拦，一个进程能对它能打开的任意 pts 这么做，不限于自己那个。历史上同一个原语就被用来逃出受限 shell、往别人的终端里塞命令。

`claude attach` 那条不碰这个特权原语，用的是官方客户端，相对温和。但它照样是把输入送进一个会话；`--session <id>` 更是显式地指向另一个会话——所以它同样只该用在你自己的机器、你自己的会话上。

工具的自动发现刻意只认"进程祖先链上最近的那个 Claude 会话"，找不到就拒绝。那是**使用层面**的自我约束，不是内核强制的边界——root 随时能绕过去（`--session` 和 `--unsafe-pts` 就是把这一点摆到明面上）。

## 几条规矩

别在多用户或共享主机上跑它。在那种机器上，一个能拿 root 的键盘注入器就是提权和会话劫持工具，这个 wrapper 自己再克制也没用。

别去碰不属于你的会话。自动发现只会指向你派生自的那个最近的 Claude；不要用 `--session` 或 `--unsafe-pts` 去打别人的终端/会话把这层保护绕掉。

别去开 `dev.tty.legacy_tiocsti=1`。现代内核把老式的 `TIOCSTI` 关掉是有意为之。这个工具靠 `CAP_SYS_ADMIN`（root）在不重新打开它的前提下工作；我们不写、也不建议把它开回来，因为那等于把注入原语暴露给没有特权的进程。

## 这个工具做了哪些加固

**只认最近的 Claude 祖先**。自动发现从进程树往上找到第一个 Claude 就地判定，绝不越级——否则一个从交互式前台会话派生出的后台会话，可能被误判、把 goal 注入到前台父会话里。

**用控制终端区分活的 pts 与 daemon 空壳**。标准输入/输出得是同一个 pts、且那个 pts 是进程的控制终端，才走 pts 注入；daemon 会话（无控制终端）改走 attach。

**daemon 会话 id 从自有的、恰好一个的 listening rv socket 读**，0 个或多个都 fail-closed，免得选错会话。

**条件净化**。goal 文字里任何控制字符（回车、换行、ESC 等）都会被拒，注入的这行没法夹带第二次按键或第二条 slash 命令。

**硬性 dry-run 保险**。设 `CLAUDE_SELF_GOAL_DRY_RUN=1` 工具就只 dry-run、绝不注入；测试套件一律带着它，任何测试即使写错也不可能注入活会话。

**越权的口子既显式又自证**。绕过自动发现要么给 `--session <id>`（跨会话），要么给 `--unsafe-pts` 加上 `--i-understand-this-can-inject-keystrokes`。

发现这工具能被用到它声明的模型之外，或者哪条保护有 bug？欢迎开 issue。这是个小而专的工具，没有单独的保密上报渠道。
